\subsection{Introduction}

This chapter goes through how the proposed \textbf{Digital Admission Script Evaluation and Moderation System (DASEMS)} was actually built. It was developed as a web platform to automate the descriptive admission script evaluation workflow from end to end.

The implementation is split into several components tied together in a modular architecture, along with a Python-based document processing service that handles PDF analysis and answer extraction. These pieces talk to each other through RESTful APIs, which keeps integration clean and the system scalable.

Altogether, the system covers admission management, PDF processing, teacher assignment, evaluation, moderation, and result generation.

\subsection{Development Environment}

\textbf{DASEMS} was built on a modern full-stack setup chosen to suit both web application development and document processing.

The frontend runs on \textbf{React 18} and \textbf{TypeScript}, and the backend is built with \textbf{Node.js} and \textbf{Express.js}. \textbf{MongoDB} was chosen as the database mainly because its document-oriented structure fits the kind of data this system deals with. A separate Python service, built on \textbf{FastAPI}, handles PDF processing, OCR, and image analysis, using \textbf{PyMuPDF}, \textbf{pdfplumber}, \textbf{Pillow}, and \textbf{Tesseract OCR}.

Docker Compose containerizes the backend, database, Redis cache, and Python processing service, so deployment stays consistent no matter where it's run. Git handled version control, and Visual Studio Code was the main development environment throughout.

The whole system was developed and tested on macOS, with Docker Compose managing communication between services to keep the development setup stable and predictable.

\subsubsection{Authentication and Authorization Module}

Authentication relies on \textbf{JWT}, with passwords hashed using \textbf{bcrypt}. Once a user logs in, a JWT gets issued and then checked on every subsequent request to a protected route.

\subsection{Backend Implementation}

Built with \textbf{Node.js} and \textbf{Express.js}, it's organized into controllers, services, repositories, models, and middleware, keeping each concern separate.

The subsections below cover the backend's structure, how authentication works, and how the REST API is put together.

\subsubsection{Project Structure}

The backend follows a feature-oriented folder structure -- related files sit together in their own module, which makes the codebase easier to maintain and simpler to extend later.

The main directories are:

\begin{itemize}
\item \textbf{Config}: Configuration and database settings.
\item \textbf{Routes}: REST API endpoints.
\item \textbf{Controllers}: Request handling.
\item \textbf{Services}: Business logic.
\item \textbf{Repositories}: Database operations.
\item \textbf{Models}: MongoDB schemas.
\item \textbf{Middleware}: Authentication, authorization, and validation.
\end{itemize}

React's component-based structure keeps the app responsive since only the components that actually change get re-rendered, and it makes reusing code across the interface much easier. React Query takes care of asynchronous API calls and caching, while Redux Toolkit keeps track of authentication and other global application state.

\subsubsection{User Interface Design}

The interface is built around dashboards, so each user only sees the functions their role allows. Once someone logs in, they're taken straight to their own dashboard automatically. Navigation stays simple throughout, relying on responsive layouts, tables, and cards rather than deep menus.

\subsubsection{Administrative Dashboard}

From the Administrative Dashboard, Super Administrators handle admission sessions, users, departments, subjects, question banks, student registration, answer script uploads, evaluation progress, and result generation.

Real-time stats, notifications, and progress indicators give administrators a single place to keep an eye on how processing is going.

\subsubsection{Teacher Evaluation Workspace}

Teachers only see the answers assigned to them, through their own evaluation workspace. Each evaluation page shows the question, the model answer, the marking rubric, the anonymous student identifier, the extracted answer image, annotation tools, a field for entering marks, and a comment section.

\subsubsection{Head Examiner Dashboard}

The Head Examiner Dashboard focuses on answers that need moderation. It shows the extracted answer image, the model answer, the marking rubric, the marks from every evaluator, their comments, and the annotation layers.

\subsubsection{Frontend Communication}

The frontend reaches the backend through RESTful APIs secured with JWT. React Query handles the asynchronous side -- requests, caching, keeping data in sync -- while Redux Toolkit manages user sessions and authentication state.

\subsection{Database Implementation}

\textbf{DASEMS} runs on \textbf{MongoDB} as its main database, largely because its document-oriented structure handles the mixed, varied nature of the data well -- users, admission sessions, question banks, students, answer scripts, evaluations, moderation records, and audit logs all fit naturally. Database operations go through \textbf{Mongoose}, which adds schema validation, object modeling, and a simpler way to work with MongoDB collections.

\subsubsection{Collection Implementation}

Each major entity in the system gets its own MongoDB collection, which keeps things easier to maintain and scale. The main ones are:

\begin{itemize}
    \item \textbf{Users}: Authentication credentials, user profiles, and roles.
    \item \textbf{Admission Sessions}: Examination configuration and session information.
    \item \textbf{Questions}: Question bank, model answers, and marking rubrics.
    \item \textbf{Students}: Student information with anonymous identifiers (\textbf{s\_code}).
    \item \textbf{Scripts} and \textbf{Answers}: Uploaded PDFs and extracted question-wise answer images.
    \item \textbf{Assignments}, \textbf{Evaluations}, and \textbf{Adjudications}: Teacher assignments, submitted marks, and moderation records.
    \item \textbf{Audit Logs}: System activities and administrative actions.
\end{itemize}

\subsubsection{Relationship Implementation}

Collections link to each other through MongoDB document references. Each student sits within an admission session and has one uploaded script tied to them. That script gets broken into multiple answers, each of which goes to three teachers for independent evaluation. If the moderation threshold gets crossed, the evaluation records get linked to an adjudication record that the Head Examiner maintains.

\subsubsection{Data Integrity and Consistency}

Mongoose schema validation checks that every document meets its constraints before it's saved. Once an evaluation is submitted, it stays immutable -- moderation decisions get stored as separate records instead, so the original evaluation history is preserved. Audit logs are append-only, which keeps a full, traceable record of everything that happens in the system.

\subsubsection{PDF Upload and Validation}

Every uploaded PDF is checked for file type and integrity first, then stored as a read-only document before anything else happens to it.

\subsubsection{PDF Rendering}

Each page gets rendered into a high-resolution image using PyMuPDF, which gives the rest of the pipeline a consistent, standardized input to work from.

\subsubsection{Question Segmentation and Answer Extraction}

Rendered pages are processed against a predefined document template to identify and segment question-wise answer regions. OCR is brought in as needed for scanned documents. Each extracted answer, along with its metadata, gets written to the database -- including the anonymous student identifier for that answer. Every segmented answer region is cropped out as its own standalone image, and the original uploaded document is never altered.

\subsection{Teacher Assignment and Evaluation Implementation}

Once the PDF processing module pulls out the question-wise answers, the backend picks up the associated metadata -- subject, question number, anonymous student identifier (\textbf{s\_code}), and where the answer image is stored.

From there, the assignment engine picks teachers based on their subject specialization and how much they're already carrying, so the workload stays reasonably balanced.

The evaluation interface pulls all of this together: the extracted answer image, the question statement, the model answer, the marking rubric, annotation tools, and a form for submitting marks. Once a teacher submits their evaluation, that record becomes read-only unless it's later touched through the moderation process.

\subsubsection{Moderation Implementation}

To keep grading consistent, the system automatically checks the marks submitted by all three evaluators against each other. If the gap between the highest and lowest mark falls within the moderation threshold, the final score is just the average of the three. If not, the Head Examiner can step in and review it directly.

\subsection{Result Generation}

Once evaluation and moderation wrap up, the system generates the final results on its own. Before it calculates a student's final mark, it first checks that every assigned evaluation for that student has actually been completed.

Results get stored in MongoDB and shown through the administrative dashboard. Alongside the final results, the system also produces evaluation reports and statistics -- total scripts evaluated, evaluations still pending, how many answers went through moderation, subject-wise performance, and teacher workload. Automating result generation this way cuts down on manual calculation errors and makes the whole evaluation process more efficient and transparent.
